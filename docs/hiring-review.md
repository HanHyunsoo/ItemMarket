# 채용 관점 냉정 리뷰 — Wasteland Exchange

> 평가자: 게임 서버 시니어(10년+) / 채용 담당 관점
> 대상 직무: **낙원: LASTPARADISE** (UE5 좀비 PvPvE 익스트랙션 서바이벌) — 서비스 시스템 설계·구축 / 서비스 로직 최적화·분석 / 운영 툴 / FE·BE. 자격: **게임 서버 5년+**. 우대: **MSA**, **문제해결**, **C#**.
> 리뷰 방식: README/문서뿐 아니라 실제 소스(`OrderBookEngine`, `MarketRepository.SettleFillAsync`, 인증/허브/엔드포인트, DDL, 테스트, CI)를 직접 읽고 검증함.

## 1. 총평 — 서류는 통과, 면접은 "도메인 미스매치"에서 갈린다

**강점 (3줄로만):** (1) 동시성/정합성 스토리가 말이 아니라 **코드+테스트+SQL 불변식**으로 증명된다 — Orleans 단일 활성화로 매칭을 직렬화하고, 정산을 단일 Postgres 트랜잭션으로 원자화하며, 동시 매수/취소·데드락·멱등성을 실제 통합 테스트로 고정했다. (2) **자체 감사(Critical 3건)와 데드락 발견→락 순서화→재측정(p99 973→175ms)** 은 5년차+에게 기대하는 "문제 정의→분석→수정→회귀"의 정석을 보여준다. (3) 코드 품질·문서화·CI·Docker 원커맨드·풀스택까지 완성도가 높아 "혼자 서비스를 굴려본 사람"이라는 인상을 준다.

**냉정한 결론:** 순수 백엔드 엔지니어링 역량은 **주니어~미들 상위, 시니어 초입** 수준으로 서류는 무난히 통과한다. 다만 이 JD는 **"게임 서버 5년+"** 이고 대상이 **UE5 좀비 PvPvE 익스트랙션 슈터**다. 이 포트폴리오는 본질적으로 **웹/메타서비스(거래소) 백엔드**이지, **게임플레이/세션/매치 서버**가 아니다. 실시간 전투 틱, 리플리케이션/네트코드, 데디케이티드 서버 오케스트레이션, 익스트랙션 세션 루프, 게임플레이 안티치트 — JD가 말하는 "서비스 시스템"의 게임적 핵심을 **직접 보여주는 코드가 없다.** 면접관은 정확히 여기를 판다. 지원자도 이걸 알고 "익스트랙션 세션 루프는 후속"이라고 방어선을 쳐뒀지만, **방어선을 쳤다는 것 자체가 갭이 있다는 자백**이다. "5년차 게임 서버"라기엔 게임 서버 특유의 문제(틱레이트, 상태 동기화, 매치메이킹, 세션 수명주기, 지연/패킷 손실)를 다룬 흔적이 없다.

한 문장 요약: **"거래소 백엔드로는 A-, LASTPARADISE 게임 서버 지원서로는 B0. 통과시키되, 면접에서 게임플레이 서버 경험을 반드시 검증해야 한다."**

### MSA "우대"인데 MSA를 안 쓴 것 — 어떻게 방어/보완하나
- **방어(면접 화법):** "Orleans의 virtual actor는 사실상 **grain 단위 마이크로서비스**입니다. 위치 투명성·단일 활성화·멤버십(클러스터링)이라는 분산 시스템의 핵심 3요소를 취하되, 브로커/디스커버리/독립 배포의 운영비용은 지불하지 않는 선택을 했습니다. 즉 MSA를 몰라서 안 쓴 게 아니라, **이 도메인(경쟁 상태가 지배적)** 에 MSA가 과하다고 판단한 겁니다." — 이 논리는 강하다. 문제는 **판단만 있고 MSA 실물이 없다**는 것.
- **보완(코드로):** 지금 구조는 이미 MSA로 쪼개기 좋다. 최소 노력으로 신뢰도를 크게 올리려면:
  1. **정산(Settlement)만 별도 워커 서비스로 분리** — 체결 이벤트를 큐(예: Postgres LISTEN/NOTIFY 또는 경량 브로커)로 던지고 별도 프로세스가 정산. "동기 정산이 grain 서비스 시간을 지배한다"는 본인 분석(`perf-report.md`)의 자연스러운 다음 수.
  2. **인증/토큰 서비스 분리** 또는 **카탈로그 읽기 서비스 분리** 정도라도 실제 프로세스 경계를 하나 그어보면 "MSA도 해봤다"가 성립.
  3. 최소한 **README에 "MSA로 전환한다면 서비스 경계·계약·일관성(사가/아웃박스) 설계"** 를 1페이지로 그려두면 "관심"은 확실히 증명된다. 지금은 이게 없다.

## 3. 약점·리스크 (면접관이 파고들 구멍, 우선순위순)

### R1. [치명적·전략] 도메인 적합성 — 이건 거래소지 게임 서버가 아니다
- **구멍:** LASTPARADISE는 UE5 실시간 PvPvE. 이 레포엔 게임 클라이언트 연동, 세션/매치 서버, 틱 기반 시뮬레이션, 상태 리플리케이션, 데디 서버 스케일링이 전혀 없다. "서비스 시스템"을 **경제/거래 메타서비스로만** 해석했다.
- **왜 위험:** 면접관 첫 질문이 "우리 게임의 인게임 서버는 UE dedicated인데, 거기서 초당 수십 틱으로 도는 상태 동기화·판정·안티치트를 다뤄본 적 있나요?"일 것. 이 레포는 답을 못 준다.
- **수정안:** (a) **익스트랙션 세션 루프**를 최소라도 구현(아래 5장). (b) 이력서/README 상단에 "이 프로젝트가 증명하는 것 = **게임 메타/서비스 백엔드**(경제·거래·인벤·정산). 인게임 시뮬레이션 서버는 [별도 경험/학습 계획]"이라고 **스코프를 정직하게 프레이밍**해 미스매치를 논쟁이 아닌 합의로 만든다.

### R2. [높음] 데모 로그인에 비밀번호 없음 + 시크릿 커밋
- **구멍:** `AuthEndpoints.cs` L23 — `GetPlayerAsync(req.PlayerId)`만으로 토큰 발급. 자격 증명 검증 0. `appsettings.json` L14에 JWT 시크릿 평문 커밋. admin은 하드코딩 playerId(`JwtTokenIssuer` L31)로 판정.
- **왜 위험:** 본인도 한계로 명시(README, `backend-audit.md` L1). 정직함은 가점이나, "인증을 설계할 줄 안다"는 증거는 리프레시 로테이션/재사용 탐지(이건 **잘 됨** — `AuthEndpoints` L37-48, `MarketRepository` L638)뿐. 면접관은 "그럼 실제 비번 해싱(Argon2/bcrypt), 비대칭키(RS256) 회전, 시크릿 매니저는?"을 물을 것.
- **수정안:** 로그인에 **Argon2id 비번 검증** 한 겹만 추가(2시간 작업). JWT를 **RS256**으로 바꾸고 공개키 노출 엔드포인트(JWKS)만 흉내내도 "무상태 검증"을 진짜로 증명. 시크릿은 `appsettings`에서 빼고 env-only로.

### R3. [높음] 관측성 부재 — "최적화·분석" 직무인데 프로덕션 관측 스택이 없다
- **구멍:** 구조적 로깅뿐. 메트릭/트레이싱/대시보드 없음(본인도 로드맵으로 인정, `interview-prep.md` L141). 성능 분석이 **자작 스크립트 + 로그**에 의존.
- **왜 위험:** JD 2번(로직 최적화·분석)의 실무는 "라이브 지표를 보고 병목을 찾는" 일. 부하도구로 발견한 건 훌륭하지만 그건 **랩 환경 재현**이지 **프로덕션 관측**이 아니다.
- **수정안:** **OpenTelemetry** 계측(정산 트랜잭션 span, grain 호출 히스토그램) + **Prometheus 메트릭**(orders/s, p99, 체결율, 데드락 카운터) + Grafana 대시보드 1장. 이미 있는 부하도구 수치를 실시간 그래프로 바꾸면 "분석 역량"이 서류에서 즉시 보인다. **노력 대비 임팩트 최상위.**

### R4. [높음] 단일 노드 성능 수치 — 절대치가 modest하고 반박당하기 쉽다
- **구멍:** 1,411 orders/s(spread), 350(hot), 649(밴딩). 전부 **M2 Pro 단일 노드, API+실로+DB+부하도구가 한 머신**(`perf-report.md` L38, 225). 지원자도 "절대치보다 상대 비교"라고 캐비어트.
- **왜 위험:** 면접관: "1,411 orders/s면 인기 게임 하나 감당 못 하는데, 실제 분산 배포에서 어디까지 가나? 단일 Postgres가 상한이라며 그 다음은?" 지금 답은 "읽기 리플리카/샤딩"이라는 **말뿐**.
- **추가 구멍(측정 방법론):** 부하도구가 **closed-loop**(워커가 자기 요청에 블로킹) + **자가 측정 p99**(`Driver.cs` nearest-rank)라 **coordinated omission**으로 **꼬리 지연을 과소평가**한다. 게다가 부하도구가 **시딩과 채점을 자기가** 한다(플레이어·템플릿당 100만 재고 → 매도 항상 성공하는 합성 best case). 즉 수치가 유리하게 나오는 조건. 면접관이 부하 코드를 읽으면 지적한다.
- **수정안:** (a) 실로/DB를 물리 분리한 **멀티노드 측정** 1회라도. (b) 부하도구를 **open-loop(도착률 고정)** 로 바꾸거나 최소한 coordinated-omission 보정. (c) 정산이 병목이면 **배치 정산/파이프라이닝**을 실제로 넣어 상한을 올려 보이기. (d) "이 수치는 아키텍처 검증용 상대 지표"라고 **더 크게** 못 박고 절대치 홍보를 줄이기.

### R5. [중] 매칭 엔진(핵심 로직)에 유닛 테스트가 없다 — 테스트가 통합에 편중
- **구멍:** 유닛 테스트는 순수 함수(`CalcFeeTests.cs`, `StashGeometryTests.cs`)뿐. **`OrderBookEngine`의 매칭 로직**(가격-시간 우선, 부분 체결, 자전거래 skip, 잔량 처리)은 전부 **DB를 띄우는 통합 테스트**로만 검증(`MarketFlowTests.cs`, `HardeningTests.cs`). 엔진이 `MarketRepository`(구체 sealed 클래스)·`IGrainFactory`에 강결합돼 있어 **격리 유닛 테스트가 구조적으로 불가능**하다.
- **왜 위험:** "컨텐츠 로직 설계"가 JD 자격. 정작 이 프로젝트의 핵심 로직이 I/O와 엉켜 단위 검증이 안 된다. 면접관: "매칭 우선순위 규칙만 딱 떼어 100개 케이스 돌리려면?" → 지금은 매번 Postcontainer를 띄워야 함(느림).
- **수정안:** `MarketRepository`를 **인터페이스(`IMarketRepository`)** 로 추출하고 에스크로/정산을 포트로 분리 → 매칭 엔진을 **인메모리 fake**로 유닛 테스트. 통합 우선 전략은 유지하되 **핫 로직은 유닛 커버**하는 게 정석. (통합 테스트 자체는 품질이 매우 높다 — Testcontainers로 실제 DB, 동시성/데드락/멱등성까지 실검증하고 **CI에서 컨테이너 포함 전 스위트가 돈다**. 부정이 아니라 **경계 보강** 지적.)
- **참고(README 숫자):** "테스트 65개"는 xUnit **실행 케이스 수**(Theory가 InlineData로 전개된 값)로는 정직하지만, **테스트 메서드 수는 53개**다. "단위 33 + 통합 28 + 밴딩 4"는 단위만 케이스 수, 나머지는 메서드 수로 세는 혼용이라 면접에서 "실제 몇 개냐"에 정확히 답할 수 있어야 한다.

### R6. [중] 정산 원자성의 "구멍" 두 개를 보상 트랜잭션으로 때움
- **구멍:** (a) **에스크로와 주문 INSERT가 별개 트랜잭션**(`OrderBookEngine` L127-177). 사이에서 죽으면 자산이 잠긴 채 주문이 없어 복구 불가 → **보상(compensation)** 으로 원복, 보상마저 실패하면 `LogCritical` 후 수동 복구(`backend-audit.md` C2). (b) **더블 활성화 시 에스크로 비멱등**(`backend-audit.md` M4) — 멱등 키 없음.
- **왜 위험:** 본인이 정직하게 문서화한 건 훌륭하나, 면접관은 "왜 처음부터 에스크로+INSERT를 한 트랜잭션으로 안 했나?"를 묻는다. 답: grain(WalletGrain)과 OrderBook이 분리돼 에스크로가 별도 grain 호출 → 별도 tx. 이건 **Orleans 액터 경계가 트랜잭션 경계와 어긋나는** 전형적 문제.
- **수정안:** 에스크로+주문 INSERT를 **하나의 DB 트랜잭션**으로 합치기(WalletGrain을 거치지 말고 정산 리포지토리에서 직접) → 보상 로직 자체가 사라진다. 더블 활성화는 `wallet_ledger(reason, ref_id)` **부분 유니크 제약**으로 에스크로 멱등화(M4 제안대로).

### R7. [중] 읽기 경로에 부수효과·비효율
- **구멍:** (a) `StashGrain.GetStash()`가 **GET인데 DB를 write**한다 — stale 배치 DELETE + 자동배치 UPSERT(`StashGrain.cs` L43,102). REST 안전성(GET 멱등·부작용 없음) 위반. (b) `LoadFootprintsAsync()`가 매 호출 **카탈로그 102행 전체 로드**(`StashGrain.cs` L173), `PlaceOrderAsync`가 매 주문 `GetTemplateAsync` 재조회(L88), 매 주문 후 `GetSnapshot()` 재발행(밴딩 ON이면 DB 팬아웃까지, `OrderBookGrain` L96). (c) `wallet_ledger` 조회 정렬 `id DESC`와 인덱스 `created_at DESC` 불일치(`backend-audit.md` L4).
- **왜 위험:** "서비스 로직 최적화" 직무인데 핫패스에 불필요한 DB 왕복이 여럿. 면접관이 코드를 읽으면 바로 보인다.
- **수정안:** 카탈로그/템플릿/fee_bps를 **메모리 캐시(TTL 또는 시작시 로드)**. GET/명령을 CQRS로 분리해 GET에서 write 제거(자동배치는 별도 command로). 스냅샷은 **증분 갱신** 또는 발행 스로틀.

### R8. [중] 자전거래 skip → crossed book 방치
- **구멍:** 본인 주문은 매칭에서 skip(`OrderBookEngine` L234). 결과로 같은 사람의 매수/매도가 **교차한 채 호가창에 공존**(`backend-audit.md` H1 인정). 스냅샷 top-of-book이 왜곡될 수 있다.
- **수정안:** skip 대신 **주문 시점 거부(자기 물량과 교차 금지)** 또는 등록 시 반대측 본인 잔존을 취소하는 정책. "게임 마켓이라 무해"는 통하지만, 시세 신호 조작 여지를 남긴 건 시장 설계자로선 약점.

### R9. [낮음] 캐시 무효화·설정 반영 부재
- `_feeBps`/`_stackable`이 grain 활성화 시 캐시 → 운영 중 수수료 변경이 살아있는 grain에 반영 안 됨(`backend-audit.md` M5). 운영 툴 직무에서 "설정 핫리로드"는 기대 기능.

### R10. [낮음] 리엔트런시 스위치가 프로세스 전역 static
- `OrderBookGrain.AllowInterleaving`이 `static volatile`(L43) — 밴딩은 **프로세스 단위 all-or-nothing**. 템플릿별로 밴드 크기를 다르게 못 준다. 영리한 트릭이나 확장성엔 한계. 면접에서 "왜 static?" 물으면 "한 프로세스=한 설정"이라 답 가능하지만 설계 냄새.

### R11. [중] grain은 얇은 pass-through — "Orleans 마스터리"라기엔 정산·락이 다 SQL에 있다
- **구멍:** `WalletGrain`(27줄)·`PlayerInventoryGrain`(30줄)·`StashGrain`은 상태를 안 갖고 **전부 `MarketRepository`로 위임하는 pass-through**다. 매칭조차 `OrderBookEngine`(POCO)로 추출돼 grain은 라우팅 껍데기. `db/orleans-clustering.sql`은 **업스트림 Orleans 9.2.1 스키마 그대로**(자작 아님, 정직하게 표기됨).
- **왜 위험:** 실제 동시성 가치는 **DB 락 순서화(`SettleFillAsync` FOR UPDATE ORDER BY player_id)** 에 있지 grain 정교함에 있지 않다. 즉 Orleans는 "단일 활성화로 매칭 직렬화"라는 **한 가지 속성만** 쓰고, persistence·streams·timers·reminders·transactions 등 Orleans의 나머지는 안 썼다. 면접관이 "Orleans의 어떤 기능을 써봤나?" 물으면 표면이 얇다. 반대로 "그래서 Orleans persistence 대신 DB 재수화를 골랐다"는 설명은 가능(장점).
- **수정안:** 익스트랙션 세션(5장)에 **Orleans reminders(레이드 타임아웃)·grain timers·grain state persistence** 를 실제로 써서 Orleans 활용 폭을 넓히면 "액터 모델을 한 트릭이 아니라 도구상자로 안다"가 증명된다.

## 4. 면접 예상 압박 질문 Top 10 (+ 현재 코드로 답 가능한지)

1. **"우리 인게임 서버는 UE dedicated다. 실시간 틱 시뮬레이션/리플리케이션/판정을 서버에서 다뤄봤나?"**
   → ❌ **구멍.** 이 레포로는 못 답한다. 별도 경험이 없으면 치명적. 정직하게 스코프 분리 + 학습 계획으로 방어.
2. **"에스크로와 주문 INSERT는 왜 한 트랜잭션이 아닌가? 보상이 또 실패하면?"**
   → ⚠️ 부분. "grain 경계 때문"이라 답 가능하나 "그럼 합쳤어야" 반박에 약함(R6). 개선하면 완벽.
3. **"1,411 orders/s는 단일 노드다. 실제 프로덕션 목표 TPS와 스케일 전략은? 단일 Postgres 상한 다음은?"**
   → ⚠️ 말로는 가능(리플리카/샤딩), 코드/측정은 없음(R4).
4. **"매칭 엔진 우선순위 규칙만 격리해 테스트하려면?"**
   → ❌ 지금 구조론 DB 없이 불가(R5). "인터페이스 추출하겠다"가 정답.
5. **"MSA 우대인데 왜 안 썼나? Orleans가 MSA와 뭐가 다른가?"**
   → ✅ 논리 강함(3장). 단 실물 없음이 약점.
6. **"관측성 없이 프로덕션에서 이 데드락을 어떻게 발견했을 건가?"**
   → ⚠️ 랩 재현은 했으나 프로덕션 관측 스택 경험 미증명(R3).
7. **"자전거래 skip으로 crossed book이 생기면 시세가 거짓말하지 않나?"**
   → ⚠️ 인지·문서화됨(H1). "게임 마켓이라 허용"은 방어 가능하나 완벽하진 않음(R8).
8. **"밴딩은 전역 가격-시간 우선을 깬다. 실제 유저가 '더 싼 매물이 있는데 왜 안 잡혔지?' 하면?"**
   → ✅ 트레이드오프 명확히 문서화·테스트(`perf-report.md`, 밴딩 회귀 테스트). 강한 답.
9. **"grain 상태를 왜 Orleans persistence(저장소)로 안 하고 매번 DB 재수화하나?"**
   → ✅ "DB가 소스오브트루스, 인메모리는 재구성 가능한 투영" 논리 명확(`OrderBookEngine` 주석). 좋음.
10. **"운영자가 특정 유저의 RMT(현금거래)를 의심한다. 지금 툴로 뭘 할 수 있나?"**
    → ⚠️ `wallet_ledger`로 수동 추적만 가능. 이상탐지/대시보드/제재 툴 없음(R1 운영 툴 얕음).

## 5. 도메인 적합성 강화안 (좀비 PvPvE 익스트랙션에 맞추기)

지금 프로젝트를 **버리지 말고**, 장르 핵심 루프에 연결하면 미스매치가 강점으로 바뀐다. 우선순위순:

1. **익스트랙션 세션 루프 (최고 임팩트).** 최소 서버 시뮬:
   - `RaidSessionGrain`(키=sessionId): 참가자·반입 인벤(로드아웃)·상태(진행/탈출/사망) 관리.
   - **반입(insert):** 스태시→세션 인벤으로 아이템 락(에스크로 재사용). **탈출(extract):** 세션 인벤→스태시 귀속. **사망(death):** 세션 인벤 소실(다른 플레이어에게 루팅 가능하게 드롭).
   - 이 루프가 **마켓 유입**과 연결(탈출 전리품을 거래소에 판매). 이미 있는 에스크로/정산/원장 인프라를 그대로 재사용 → **적은 코드로 "게임 서비스 시스템"** 을 증명.
2. **매치/세션 관리 서비스.** 세션 생성·정원·수명주기·타임아웃(레이드 제한시간)·재접속 처리. Orleans grain 수명주기와 잘 맞는다. "세션 서버 이해"를 증명.
3. **이상탐지/안티치트(경제).** append-only `wallet_ledger`가 이미 있으니 그 위에 **RMT/봇 탐지 규칙**(비정상 이체 패턴, 자전거래 클러스터, 급격한 자산 증가)과 **자동 플래그 + GM 리뷰 큐**. JD "운영 툴"에 정확히 꽂힌다.
4. **운영(GM) 대시보드.** 실시간 경제 지표(발행량/소각량/거래량/가격 인덱스), 유저 조회/제재(밴·롤백·아이템 회수), 감사 로그. 지금 admin 6개 엔드포인트를 **진짜 운영 콘솔**로 승격.
5. **인벤토리 영속화·서버 권위.** 스태시는 이미 서버 권위 검증(겹침·경계)이 있음 — 이걸 **세션 인벤/사망 드롭/전리품 롤백**으로 확장하면 "치트 방지 서버 권위"를 게임 문맥에서 증명.
6. **그리드 확장:** 회전(rotation)·다중 컨테이너·무게/부피 — 익스트랙션 장르의 인벤 UX 핵심(지원자도 언급). FE·BE 양쪽 역량 재증명.

## 6. 실행 권고 Top 5 (노력 대비 채용 임팩트순)

1. **익스트랙션 세션 루프 MVP 추가 (임팩트 최상 / 노력 중상).** `RaidSessionGrain` + 반입/탈출/사망 → 마켓 유입. **R1(도메인 미스매치)을 정면 해소**하고, 기존 에스크로/정산/원장을 재사용하므로 코드 대비 임팩트가 압도적. 이거 하나로 "거래소 개발자"가 "게임 서비스 개발자"로 바뀐다.
2. **OpenTelemetry + Prometheus + Grafana 대시보드 1장 (임팩트 상 / 노력 하).** 이미 있는 부하도구 수치를 실시간 그래프로. **R3(관측성)** 를 지우고 JD "최적화·분석"을 서류에서 즉시 증명. 반나절~하루.
3. **README/이력서 스코프 재프레이밍 (임팩트 상 / 노력 최하).** "이 프로젝트 = 게임 **메타/서비스** 백엔드(경제·거래·정산·인벤). 인게임 시뮬 서버는 [세션 루프로 확장 중/별도 학습]"이라고 상단에 명시. **R1을 논쟁이 아닌 합의**로. 30분.
4. **MSA를 실제로 한 조각 쪼개기 (임팩트 중상 / 노력 중).** 정산 워커를 별도 프로세스+큐로 분리하거나, 최소 "MSA 전환 설계 1페이지". **MSA 우대 항목**을 "관심"에서 "경험"으로. 본인 성능 분석의 자연스러운 후속이라 스토리도 매끄럽다.
5. **매칭 엔진 유닛 테스트화 (임팩트 중 / 노력 중).** `IMarketRepository` 인터페이스 추출 → 인메모리 fake로 매칭 규칙 격리 테스트. **R5** 해소 + "핫 로직은 유닛, 통합은 계약검증"이라는 성숙한 테스트 피라미드를 보여줌. (덤으로 R6 에스크로+INSERT 트랜잭션 통합도 같이 정리 가능.)

### 부록 — 검증하며 확인한 "실제로 잘 된 것" (면접에서 밀어야 할 무기)
- 정산 락 순서화가 **취소 경로에도 일관 적용**됨(`MarketRepository.CancelOrderAsync` L496 주석+구현) — 데드락 수정이 말뿐이 아님.
- 리프레시 토큰 **원자적 로테이션 + 재사용 탐지 체인 폐기**가 실제 구현·테스트됨(`AuthEndpoints` L37-48, `AuthTokenTests`).
- 멱등성이 **claim→처리→저장, 실패 시 슬롯 해제, 처리중 409**까지 상태기계로 제대로 구현(`ApiResults.ExecIdempotent` L71).
- 정합성 불변식(`발행=지갑+에스크로+소각`, 아이템 보존, 음수잔액 0)을 **부하 후 SQL로 실검증**(`perf-report.md`).
- 밴딩의 "inherent serialization" 통찰(전역 가격-시간 우선은 단일 직렬화 지점을 원리적으로 요구)은 **시니어급 사고**.
