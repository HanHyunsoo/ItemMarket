# 면접 핫시트 — 시니어 Orleans 게임서버 면접관이 던질 질문

> **관점**: Orleans 기반 게임 서버를 실제로 설계·운영해 본 시니어가 이 저장소를 30분 훑고 던질 법한
> 질문을 모았다. 소프트볼이 아니라 **"진짜 이해했나 vs 카고컬트인가"를 가르는** 질문 위주다.
> 각 항목은 **Q(질문) · 무엇을 보나(probe) · 답변 방향/근거**로 구성했다. `[갭]`은 이 프로젝트가
> 의도적으로 안 한 것 — 면접에서 **먼저 인정하고 방향을 말하면** 오히려 가점이다.
>
> 짝 문서: [`interview-prep.md`](interview-prep.md)(내 답변 스크립트·STAR), [`backend-audit.md`](backend-audit.md)(자체 감사), [`perf-report.md`](perf-report.md)(실측).

## 0. 가장 날카로운 5개 (여기서 깊이가 갈린다)

1. **"정산 DB 쓰기를 그레인 턴 안에서 동기로 한다. 그럼 그 그레인은 매 주문마다 DB 왕복 동안 블록되고, 그게 단일 종목 처리량 상한(≈350 orders/s)이다. 실제 매칭 엔진은 인메모리로 마이크로초에 매칭하고 저널을 비동기로 쓴다 — 왜 DB를 크리티컬 패스에 뒀나? 어떻게 걷어낼 건가?"**
   - **보는 것**: 정확성-지연 트레이드오프를 *의식적으로* 골랐는지, 대안(이벤트 소싱/비동기 저널)의 비용을 아는지.
   - **답 방향**: "DB=진실 + 단일 트랜잭션 정산"으로 **정확성을 먼저** 샀다(포트폴리오 목표). 인메모리 매칭 + append-only 저널로 DB를 핫패스에서 빼면 처리량이 오르지만, 크래시 시 저널 재생·정합성 재구축·outbox류 배달 보장이 필요해진다 — 그 복잡도를 **의도적으로 뒤로 미뤘다**(outbox를 실제로 검토 후 스코프에서 제외). 지금 구조는 재수화 테스트로 무손실을 증명했고, 다음 단계는 정확히 이 비동기 저널화다.

2. **"에스크로는 `WalletGrain.TryEscrow`(그레인 직렬화)로 하는데 체결 정산은 `MarketRepository`가 지갑 행에 직접 SQL(`FOR UPDATE`)을 쏜다. 같은 지갑에 동시성 제어 메커니즘이 둘이다. 안 부딪히나? 정합성 경계가 그레인인가 DB인가?"**
   - **보는 것**: "그레인 단일 활성화"를 정합성 근거로 착각하는지, 진짜 경계가 **DB 행 락**임을 아는지.
   - **답 방향**: 정합성 경계는 **DB**다. `WalletGrain`은 상태를 캐시하지 않는 패스스루라 스테일 인메모리가 없고, 정산은 `SELECT … FOR UPDATE ORDER BY player_id`로 지갑 행을 잠근다. WalletGrain의 단일 활성화는 에스크로 요청을 플레이어별로 직렬화하지만 **최종 심판은 DB 락**이다(그래서 정산이 그레인을 안 거쳐도 안전). 이 이중성은 무해하나, "그레인이 정합성을 준다"는 서술은 틀렸음을 안다.

3. **"인메모리 호가창을 '투영'이라 부른다. 정산 트랜잭션 COMMIT과 그레인 인메모리 반영은 별개 단계다. COMMIT 직후·인메모리 반영 직전에 실로가 죽으면?"**
   - **보는 것**: 파생 상태의 재구성 가능성·재수화 경로를 진짜 이해하는지.
   - **답 방향**: 인메모리는 버리는 캐시다. 다음 활성화에서 `OnActivateAsync`가 DB(미체결 주문)에서 **재수화**하므로 무손실. `CrashRecoveryTests`가 **강제 비활성화 → DB-only 주문이 재수화 스냅샷에 나타남**으로 증명. DB가 최종 진실.

4. **"단일 종목 핫 그레인을 가격밴드로 샤딩하면서 '전역 가격-시간 우선'을 포기했다. 실거래소에선 그거 못 판다. 우선순위를 지키면서 병렬화할 방법은?"**
   - **보는 것**: 원리적 한계("단일 종목 엄격 우선 = 단일 직렬화 지점")를 아는지, 샤딩이 뭘 깨는지 아는지.
   - **답 방향**: 단일 종목의 엄격한 가격-시간 우선은 **본질적으로 단일 시퀀서**를 요구한다 → 병렬화하려면 우선순위 완화(밴드 격리)를 감수해야 하고 이건 문서화된 opt-in 트레이드오프. 진짜 병렬화는 "샤딩"이 아니라 **매칭을 인메모리로 극도로 빠르게 + 지속화를 비동기로**(1번)가 정답. 밴드 샤딩은 "느슨해도 되는" 종목에만.

5. **"주문 등록에서 에스크로 커밋과 주문 INSERT가 별개 트랜잭션이다. 그 사이 프로세스가 죽으면 고아 에스크로가 남는다(직접 audit에 적어놨더라). 프로덕션에서 이걸 어떻게 완전히 닫고, 이미 생긴 고아는 어떻게 찾나?"**
   - **보는 것**: 자기 시스템의 실패 창을 아는지(정직성), 실운영 복구 감각.
   - **답 방향 `[갭]`**: 현재는 보상 트랜잭션 + `OrderExists` 멱등 재조정으로 *완화*(L9a)만 했다. 완전 해결은 (a) 에스크로+INSERT를 **동일 트랜잭션**으로, 또는 (b) **transactional outbox/사가**로 원자화. 고아 탐지는 `wallet_ledger`의 `ORDER_ESCROW` vs 살아있는 주문 대사(reconciliation) **배치 잡** — 아직 없음(다음 스텝).

## 1. Orleans 그레인 모델 · 동시성

- **Q. `OrderBookGrain`이 non-reentrant인데 턴 안에서 `await SettleFill`(DB I/O)을 한다. `await` 동안 Orleans가 이 그레인에 다른 호출을 끼워 넣나?**
  - 보는 것: 리엔트런시 개념 정확도. 답: **아니다** — non-reentrant는 `await` 경계에서도 인터리브 안 함. 그래서 매칭이 안전. 대가는 I/O 동안 그레인이 놀지 못하는 것(=처리량 상한, 1번과 연결).
- **Q. 리엔트런트로 바꾸면 처리량이 오를까? 뭐가 깨지나?**
  - 답: I/O 대기 중 다음 주문을 받아 처리량은 오르나, **매칭 직렬성이 깨져** 이중 체결·경쟁이 생긴다. 그래서 안 한다. 병렬화는 리엔트런시가 아니라 인메모리 매칭+비동기 저널로.
- **Q. 매칭 로직은 그레인에 있나 리포지토리에 있나? 같은 잔주문을 두 매칭이 이중 체결 못 하게 하는 최종 방어는?**
  - 답: 매칭은 `OrderBookEngine`(그레인 소유), 정산은 repo. 최종 방어는 정산 UPDATE의 **낙관적 가드** `WHERE remaining_quantity = @before AND status IN(OPEN,PARTIALLY_FILLED)` — rows≠1이면 롤백(`OrderAlreadyClosed`). 이중 활성화가 나도 DB가 심판.
- **Q. 그레인 사전검증과 repo 변이가 별도 커넥션/트랜잭션이다(스태시 경로). TOCTOU 아닌가?**
  - 답 `[설계 인지]`: 단일 활성화에 안전성을 의존한다(qa-report 명시). 교차-그레인 경로(StashGrain↔RaidSessionGrain)는 `pg_advisory_xact_lock(playerId)`로 닫았다(F-1/F-1b). reentrant 전환 시 위험.

## 2. 정합성 · 트랜잭션 · 투영

- **Q. 정산이 한 트랜잭션에서 건드리는 테이블과 그 순서, 락 순서는?**
  - 답: `trade` INSERT → `wallet`(지갑 행 `ORDER BY player_id FOR UPDATE` 선점) → `item_instance`/`inventory_stack` → `market_order` 2건(낙관적 가드). 락 순서 통일로 40P01 데드락 제거(STAR).
- **Q. 부분 체결 + 가격 개선(테이커 상한 > 메이커가) 시 캡 흐름을 증명해봐라.**
  - 답: `gross(=메이커가×수량)`은 판매자, `improvement(=(상한−메이커가)×수량)`은 매수자 환불, 수수료만 소각. `gross+improvement = releasedEscrow = 상한가×수량` → 보존.
- **Q. 정수 절삭 수수료가 값을 새게 하지 않나?**
  - 답: 판매자 차감액 == 소각액(같은 `fee`값) → 합 0. 단위 테스트로 증명.

## 3. 직렬화 · 버저닝

- **Q. 그레인 인자/반환은 실로 경계에서 어떻게 직렬화되나? `ItemMarket.*`를 JSON으로 두는 비용은? 왜 DTO에 `[GenerateSerializer]`를 안 붙였나?**
  - 답: Contracts가 공유 계약이라 Orleans 의존을 못 넣어 **JSON 직렬화기 폴백**(`OrleansHosting`). 비용은 생성 코덱 대비 크기/속도. 단일 co-host는 in-proc copier라 영향 적음.
- **Q. `DomainException`을 `[GenerateSerializer]`로 바꿨던데, 왜 JSON 대상에서 예외를 제외했나? StackTrace/InnerException은 경계 너머로 어떻게 되나?**
  - 답: JSON은 무인자 생성자·set 없는 `Exception`을 역직렬화 못 한다 → 제외하고 생성 코덱이 `Code`를 보존(M1 수정). StackTrace는 Orleans 예외 처리가 문자열로 보존, 커스텀 필드는 `[Id]`로.
- **Q. JSON 페이로드의 **버전 내성**은? DTO에 필드 추가하고 구 실로와 신 실로를 롤링 배포하면?**
  - 답 `[갭/트레이드오프]`: System.Text.Json은 미지 필드 무시로 어느 정도 관대하나, Orleans 생성 코덱의 `[Id]` 기반 버전 관리보다 약하다. 무중단 롤링 업그레이드가 요구되면 계약 DTO에 코덱을 붙이는 하이브리드가 정석.

## 4. 스케일아웃 · 핫 그레인

- **Q. 스케일아웃을 실로 추가로 말하는데, 실측은 단일 노드다. 멀티실로에서 처음 무너지는 건?**
  - 답 `[갭]`: 최종 병목은 **단일 Postgres**. 다음은 읽기 리플리카(호가/체결/캔들 조회 오프로드) → 쓰기 샤딩. grain 분산은 되지만 정산 쓰기가 한 DB로 수렴.
- **Q. 크로스-실로에서 그레인 호출 지연·직렬화 비용이 정산 지연에 얹힌다. 지역성(locality)은 어떻게?**
  - 답: OrderBook→Wallet 호출이 실로를 넘으면 왕복이 붙는다. 정산은 지갑을 그레인 대신 직접 DB로 쳐서 grain-hop을 줄였다(2번의 이중성 이유 중 하나). 완전 최적화면 플레이어/종목 grain 배치 힌트(placement) 고려.
- **Q. 가격밴드 코디네이터를 조건부 reentrant로 뒀다. 왜? 새 병목 안 되나?**
  - 답: 코디네이터가 무상태 팬아웃이라 non-reentrant면 그 자체가 직렬 병목. 조건부 reentrant로 스냅샷 병합을 병렬화. 상태가 없어 인터리브가 안전.

## 5. 장애 · 복구

- **Q. 실로가 정산 트랜잭션 도중 죽으면?**
  - 답: 트랜잭션은 원자 → COMMIT 전이면 전부 롤백, 후면 전부 반영. 부분 상태 0. 인메모리 호가창은 재활성화 시 재수화.
- **Q. 유휴/메모리 압박으로 매칭 중 그레인이 비활성화돼도 안전한가? 증명은?**
  - 답: `CrashRecoveryTests`가 `IManagementGrain.ForceActivationCollection`으로 강제 비활성화 후 DB에서 재수화·매칭 지속·돈 보존을 검증.
- **Q. 멱등성: 같은 `Idempotency-Key`로 두 번 오면? Redis 없으면?**
  - 답: Redis 저장소로 (player,key) 슬롯 청구 → 원본 실행·중복은 저장 응답 반환. 무저장 폴백일 땐 **503으로 거부**(M2 수정) — "광고하면서 보호 0"을 제거.

## 6. 돈/아이템 회계 모델

- **Q. fungible(탄약)엔 per-unit UUID를 안 붙였다. 그럼 스택형의 프로비넌스/복제 감사는 어떻게?**
  - 답: append-only `item_ledger`(부호 delta 합 = 보유량). 유니크만 인스턴스 UUID + `origin`(SEED/RAID/RAID_LOST…). "지폐 일련번호" 비유.
- **Q. `발행 = 지갑 + 에스크로 + 소각` 불변식을 어떻게, 언제 검증하나?**
  - 답: 부하 후 SQL로 대사(perf/audit). 상시 검증하는 **리컨실 잡은 없음** `[갭]` — 프로덕션이면 주기 배치 + 알림.
- **Q. 병뚜껑 무한발행 취약점 스토리 — 정확히 뭐였고 왜 에스크로 검사를 통과했나?**
  - 답: `단가×수량` long 오버플로 → 음수 → `잔액<금액` 통과 + `잔액−(음수)`가 돈 발행. `Int128` 검증+상한으로 차단(STAR).

## 7. 실시간 (SignalR / Redis)

- **Q. 체결 후 SignalR 발행이 트랜잭션 밖 best-effort다. 커밋 후·발행 전 죽으면 클라가 이벤트를 놓친다. 트레이딩 UI에서 괜찮나?**
  - 답 `[의도적]`: 이벤트는 **진실이 아니라 "재조회 힌트"**(WalletChanged는 페이로드도 없음). DB가 진실이라 다음 조회/네비게이션에서 자가 복구. at-most-once 수용 — 이걸 outbox로 보장하는 건 자가치유되는 UI 힌트엔 오버엔지니어링이라 **의식적으로 제외**. (놓치면 안 되는 다운스트림이 생기면 그때 outbox.)
- **Q. Redis 백플레인에서 인스턴스 간 이벤트 순서 보장은? `TradeExecuted`가 `OrderBookUpdated`보다 먼저 올 수 있나?**
  - 답 `[갭]`: 강한 전역 순서 보장은 없다. 클라가 스냅샷 재조회로 수렴하도록 설계. 엄격 순서가 필요하면 시퀀스 번호/버전 부여.

## 8. 운영 · 관측성 · 배포

- **Q. 메트릭/트레이싱이 없다. 프로덕션에서 어느 그레인이 핫인지, 정산 p99가 나빠지는지 어떻게 아나? 첫 대시보드는?**
  - 답 `[갭]`: 현재 구조적 로깅만. 첫 계측은 **정산 핫패스 지연 히스토그램 + 종목별 처리량 + 그레인 활성화 수**(OpenTelemetry). 로드맵 명시.
- **Q. 활성 그레인이 거래 중인데 신 버전 무중단 배포는? in-flight 주문은?**
  - 답 `[갭]`: 현재 롤링 업그레이드 전략 미검증. Orleans 그레인 마이그레이션 + 계약 버전 내성(3번)이 전제. 정산이 원자라 배포 중 죽어도 정합성은 유지.
- **Q. 레이트리밋은 무엇 기준(per player/IP)? 스크립트 마켓메이커가 호가창을 굶기면?**
  - 답: per-player 토큰버킷(600/10s). 봇 스팸 억제용. 시장 조작 방지는 별도(호가 스푸핑 탐지 등)로 확장 필요 `[갭]`.

## 9. 트레이드오프 정면 반박 ("왜 X 아니고?")

- **Q. 왜 전용 매칭 스레드 + Kafka가 아니라 Orleans?** → 동시 주문 경쟁을 단일 활성화로 락 없이. 브로커/디스커버리 운영비 회피. 단 처리량은 1번 한계.
- **Q. 왜 실로와 API를 co-host?** → 데모 단순성. 프로덕션이면 분리해 독립 스케일. config 스왑으로 adonet 클러스터링 가능.
- **Q. 왜 Dapper인가, EF는?** → 정산 핫패스의 `FOR UPDATE`·낙관적 가드를 SQL로 명시 통제. CRUD가 늘면 그 부분만 EF 하이브리드.
- **Q. 왜 호가창을 Redis에 안 두나?** → 호가창은 `market_order`의 재구성 가능한 투영. 돈이라 ACID·내구성 필요 + 정산과 같은 트랜잭션. Redis에 두면 dual-write·캐시무효화만 늘고 재구성은 드물고 싸다. Redis는 휘발성 OK인 백플레인·멱등성에만.

## 10. 게임 도메인 특화 (Orleans 게임서버 면접관)

- **Q. 이건 아웃게임(옥션하우스)이다. 인게임 실시간 거래/세션이면 설계가 어떻게 바뀌나?**
  - 답: 세션 그레인(플레이어/파티), 재접속·상태 복원, 지연 예산(수십 ms), 클라 예측/서버 권위 조정. 현재 `extract`/`die`가 명시적 계약인 것도 "게임 서버가 결과를 알린다"는 경계 설계.
- **Q. RMT/작업장·자전거래·담합을 이 원장으로 어떻게 탐지?**
  - 답 `[확장]`: `wallet_ledger`+`item_ledger`+`trade`로 그래프 분석(반복 상대·비정상 가격·순환 이전). 현재 자전거래는 매칭에서 스킵하나, 다계정 담합 탐지는 미구현.
- **Q. 봇/스크립트 남용(무제한 마켓메이킹, 스나이핑)은?**
  - 답: 레이트리밋 + 벤더 스프레드로 완화. 정교한 조작 탐지는 로드맵 `[갭]`.

## 부록 — 먼저 인정하고 들어갈 것 (disclosed gaps, 자책골 방지)

면접관이 파기 전에 **내가 먼저** 말하면 "자기 시스템을 안다"는 신뢰가 된다:

1. **정산 DB 쓰기가 핫패스** → 처리량 상한. 다음은 인메모리 매칭 + 비동기 저널.
2. **에스크로 커밋 ↔ 주문 INSERT 별개 트랜잭션** → 고아 에스크로 창(보상으로 완화, 완전 해결은 동일-Tx/outbox + 리컨실 잡).
3. **단일 노드 측정** → 절대치보다 상대비교·불변식이 산출물. 멀티실로 벤치 미실시.
4. **관측성(메트릭/트레이싱) 없음** → OpenTelemetry 로드맵.
5. **실시간 at-most-once** → 자가치유 UI 힌트라 수용(트레이딩 체결 자체는 DB 진실).
6. **JSON 직렬화 버전 내성 약함** → 무중단 롤링엔 계약 코덱 하이브리드 필요.
7. **초점**: 아웃게임 거래소에 집중, 레이드는 얇은 경제 엔진, 게임성은 의도적 스코프 아웃.

> 핵심 태도: 이 프로젝트의 값은 "다 만들었다"가 아니라 **"뭘 왜 안 했는지 안다 + 정확성을 측정·증명했다"**. 위 갭들은 약점이 아니라 **판단의 증거**로 프레이밍한다.
