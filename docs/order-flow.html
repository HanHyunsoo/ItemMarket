주문 체결 플로우

매수 주문 하나가 호가창의 매도와 체결되기까지 — 요청부터 실시간 반영까지의 전 과정. 매칭 결정은 Orleans grain 메모리에서, 자산 이동 확정은 Postgres 트랜잭션에서.

Client API Orleans Grain(메모리) PostgreSQL Realtime
1
Client
주문 등록 요청

POST /api/orders {side:Buy, templateId:95, price, qty} + JWT (선택 Idempotency-Key)

2
API · Minimal API
인증 · 멱등성 · 라우팅

JWT의 sub로 플레이어 식별. Idempotency-Key면 Redis SET NX EX로 선점(중복 차단) → GetGrain<IOrderBookGrain>(95).PlaceOrder(...)

3
Orleans
단일 활성화로 라우팅

95번 OrderBookGrain은 클러스터 전체에 활성화 1개 + 턴 기반 → 동시 주문이 와도 한 줄로 직렬화(락 코드 0). 첫 활성화면 DB에서 미체결 주문 재수화.

4
Grain
검증 · 에스크로(자산 잠금)

상한 검증(Int128 오버플로 방어) → 매수 대금을 WalletGrain.TryEscrow로 잠금(지갑 행 FOR UPDATE). 주문 행 INSERT(OPEN) — 실패 시 보상 환불.

5
Grain · 인메모리 호가창
매칭 루프

반대편(매도) 호가를 가격-시간 우선으로 스캔, 자전거래 스킵, 교차하는 만큼 체결. 체결가 = 메이커(대기 주문) 가격. 매칭은 DB가 아니라 grain 메모리에서.

6
PostgreSQL
정산 = 단일 트랜잭션 (체결 1건당)

SettleFillAsync — 모든 자산 이동을 all-or-nothing으로:

BEGIN — one transaction
  1. 지갑 2행 FOR UPDATE (playerId 오름차순 → 데드락 방지)
  2. 판매자 대금 지급 (총액 − 수수료), 수수료 소각(sink)
  3. 아이템 이전 (스택 수량 / 인스턴스 owner → 매수자)
  4. 매수 차익 환불 (상한가보다 싸게 체결 시)
  5. trade + wallet_ledger 기록, 양쪽 주문 remaining/status 갱신
  6. 낙관적 가드 WHERE remaining = … → 이중 활성화 방어 (DB가 최종 심판)
COMMIT ✓실패 → ROLLBACK 전체

↳ 커밋 성공 후에만 인메모리 호가창 잔량 반영 — “인메모리는 재구성 가능한 투영, DB가 진실”.

7
Realtime · SignalR
실시간 발행

엔드포인트가 발행: OrderBookUpdated(새 스냅샷)·TradeExecuted → tmpl:95 그룹, WalletChanged → 매수·매도자. 다중 인스턴스는 Redis 백플레인이 중계.

8
Client
응답 · 실시간 반영

PlaceOrderResult(체결 fills[] + 잔여 주문) 즉시 수신. 다른 구독자는 폴링 없이 SignalR로 호가창·체결·지갑 갱신.

직렬화

Orleans 단일 활성화

종목당 grain 1개·턴 기반으로 매칭 경쟁을 앱 계층에서 제거 — 락 없이 직렬화.

원자성

단일 Postgres 트랜잭션

대금·수수료·아이템·원장·주문을 all-or-nothing. 실패 시 전체 롤백.

내구성

DB = 소스 오브 트루스

커밋 후에만 인메모리 반영. grain 장애 시 미체결 주문 재수화.
