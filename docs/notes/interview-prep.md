# 면접 준비 — Wasteland Exchange (게임 서비스 백엔드)

이 프로젝트로 게임 서버 백엔드 직무 면접을 볼 때의 **예상 질문 + 답변 스크립트**.
숫자·설계는 전부 실제 코드/측정 기준. 외우기보다 "왜 그렇게 했는지"의 논리를 체화하는 용도.

---

## 0. 30초 엘리베이터 피치

> "아포칼립스 익스트랙션 슈터 세계관의 **아이템 거래소**를 게임 서비스 백엔드로 만들었습니다.
> C#/.NET 10 + **Microsoft Orleans**(액터 모델)로 주문서 매칭 엔진을 구현했고, 체결은
> **단일 Postgres 트랜잭션으로 원자 정산**합니다. 실시간 호가는 SignalR, 다중 인스턴스는 Redis
> 백플레인·Orleans 클러스터링으로 확장합니다. 거래소가 진공에서 놀지 않도록 **익스트랙션 레이드 루프**를
> 아이템의 **공급원·소각처(경제 엔진)** 로 얇게 붙였습니다 — 출격 시 at-risk 잠금 → 탈출 시 제자리
> 복원/사망 소실을 **세션 상태머신 + 원자 정산**으로. 직접 만든 **부하 테스트로 데드락을 발견해 p99를
> 5.5배 개선**했고, 핫 종목 병목은 가격밴드 샤딩으로 2.2배 뚫었습니다. 통합·단위 테스트 125개로 동시성
> 정합성(돈·아이템 보존)을 증명합니다."

핵심 3키워드: **매칭엔진 · 원자정산 · 동시성 정합성(테스트로 증명)**.

## 2. 기술 선택 Q&A

**Q. 왜 Orleans인가? 왜 MSA가 아닌가?**
> 거래소의 최대 난제는 "같은 아이템에 동시 주문 N건"의 경쟁 상태입니다. Orleans는 grain
> 활성화를 **클러스터 전체에서 정확히 1개 + 턴 기반**으로 보장해서, 아이템별 매칭이 **락 코드 없이**
> 직렬화됩니다. MSA로 갔다면 브로커·서비스 디스커버리·배포 파이프라인 운영 비용을 지불하면서도
> 이 동시성 문제는 따로 풀어야 합니다. Orleans는 분산의 핵심(단일 활성화·위치 투명성·멤버십)만
> 취하고 운영 복잡도는 낮춥니다. 실제로 대형 AAA 게임 백엔드에서 검증된 프레임워크입니다. 스케일아웃은
> 실로를 늘리면 서로 다른 아이템 grain이 자연히 분산됩니다.

**Q. 정산에 왜 Orleans Transactions가 아니라 DB 트랜잭션?**
> 정산이 건드리는 대상(주문·지갑·인벤·체결·원장)이 전부 한 Postgres 안에 있어서 분산 트랜잭션이
> 필요 없는 구조입니다. Orleans Tx는 트랜잭셔널 상태 프로바이더를 요구하고 오버헤드가 있어서,
> **단일 DB 트랜잭션 + 낙관적 동시성 가드**가 더 단순하고 검증 가능합니다. 돈이 걸린 핫패스라
> "실행되는 SQL이 눈에 보이는" 통제력이 중요했습니다.

**Q. 왜 Dapper(네이티브 SQL)? EF Core는?**
> 정산은 5개 테이블을 한 트랜잭션에 묶고 `FOR UPDATE` 락 순서·`WHERE remaining=…` 낙관적 가드에
> 의존합니다. Dapper는 SQL이 그대로 보여 이런 미세 제어와 감사가 됩니다. EF의 change tracker는
> 이 경로에선 오히려 SQL을 흐리게 합니다. 도메인이 커져 CRUD 엔티티가 많아지면 그 부분만 EF를
> 얹는 하이브리드가 정석이라고 봅니다.

**Q. Redis는 왜 넣었나?**
> 단일 인스턴스에선 안 씁니다. **다중 인스턴스에서 SignalR 실시간 푸시가 인스턴스 경계를 넘게**
> 하려고 백플레인으로 도입했습니다. 클라이언트가 붙은 인스턴스와 grain 소유 실로가 다를 수 있어서,
> 백플레인이 없으면 다른 인스턴스 구독자에게 이벤트가 안 갑니다. ON/OFF 대조 실험으로 증명했습니다.
> config 스왑이라 기본은 인메모리 단일 인스턴스입니다.

**Q. 인증은?**
> JWT(HS256), `sub` 클레임으로 플레이어 식별(헤더 스푸핑 불가). 짧은 액세스(15분) + 긴 리프레시
> (14일)를 **해시로 저장·로테이션·재사용 탐지**, 프론트는 401 시 자동 리프레시 후 원요청 1회
> 재시도(single-flight). 어드민은 롤 클레임으로 인가(없으면 403).

## 3. 동시성 · 정합성 Q&A (핵심)

**Q. 한 아이템에 동시에 여러 매수가 오면 중복 체결 안 나나?**
> Orleans 단일 활성화 + non-reentrant라 그 아이템의 `OrderBookGrain`은 요청을 **한 줄로** 처리합니다.
> "단일 재고 1개에 서로 다른 7명 동시 매수 → 정확히 1건만 체결, dupe·이중지불 0"을 통합 테스트로 고정했습니다.
> 락을 직접 짠 게 아니라 액터 모델의 직렬성으로 얻은 겁니다.

**Q. 인메모리 호가창인데 서버가 죽으면?**
> 인메모리는 **"재구성 가능한 투영(projection)"**입니다. 모든 변경은 같은 트랜잭션으로 Postgres에
> write-through 되고, grain 활성화 시(`OnActivateAsync`) 미체결 주문을 DB에서 읽어 재수화합니다.
> 실로 장애·리밸런싱·유휴 비활성화 후에도 유실이 없습니다. 정산이 실패하면 grain을 버리고
> (`DeactivateOnIdle`) 다음 요청에서 DB 기준으로 다시 만듭니다 — DB가 최종 진실입니다.

**Q. 돈/아이템이 새거나 사라지지 않는 걸 어떻게 보장하나?**
> 매수는 대금을, 매도는 재고/인스턴스를 **주문 시점에 에스크로(잠금)** 해서 이중판매·복제를 원천
> 차단합니다. 체결은 단일 트랜잭션에서 {판매대금 지급, 수수료 소각, 아이템 이전, 차익 환불, 원장
> 기록, 주문 갱신}을 all-or-nothing으로 처리합니다. 부하 테스트 후 **SQL 불변식**으로 검증했습니다:
> `발행 총량 == 지갑 + 에스크로 + 소각 수수료`(diff 0), 아이템 수량 보존, 음수 잔액 0 — 수만 건
> 동시 체결에서도 전부 PASS.

## 4. "문제 해결" 스토리 (STAR) — 면접 하이라이트

**① 병뚜껑 무한 발행 취약점 (Critical)**
- 상황: 자체 감사 중 발견. `단가 × 수량`이 `long` 오버플로로 **음수**가 되면, 에스크로 검사
  `잔액 < 금액`을 통과하고 `잔액 - (음수)`가 지갑에 돈을 꽂았습니다. 경제를 즉사시키는 실익스플로잇.
- 행동: `Int128`로 곱을 계산해 상한 검증(단가 ≤ 1e12, 수량 ≤ 1e6, 총액 ≤ 1e15) 후 진입 차단,
  수수료 계산도 Int128로 통일.
- 결과: 극단값 주문이 거부되고 지갑 1캡도 안 변함을 **회귀 테스트**로 고정.

**② 부하 테스트로 데드락 발견 → 수정 → 재측정 (최고 어필)**
- 상황: 직접 만든 부하 도구로 측정하니 spread 시나리오 p99가 973ms로 튀고 간헐적 500 발생.
- 분석: 서로 다른 아이템 grain의 동시 정산이 **같은 지갑 행을 서로 다른 순서로 잠가** Postgres
  데드락(40P01). (정합성은 롤백으로 유지됐지만 재수화 왕복이 꼬리를 만듦.)
- 행동: 정산 트랜잭션 시작에 지갑 행을 **player_id 오름차순으로 일관 선점**(당시 `ORDER BY player_id
  FOR UPDATE`; 이후 락 획득이 플랜에 의존하지 않도록 C# 정렬 + 개별 문장으로 하드닝).
- 결과: p99 **973 → 175ms(5.5배)**, 데드락 0, 정합성 그대로. → "락 순서화"라는 교과서 원리를
  실측으로 발견·검증한 사례.

**③ 그리드 유니크 제약 버그**
- 같은 무기 2정을 배치하면 `duplicate key`. 원인은 초기 `(player,template)` 유니크가 인스턴스 행에도 적용된 것.
  **셀 단위 유일성**(`uq_stash_cell` + 인스턴스 전역 유일 `uq_stash_instance`)으로 재설계해 다중 스택을
  허용하면서 인스턴스 중복도 막음 + 회귀 테스트.

## 5. 심화 · 설계 판단 Q&A

**Q. 이거 결국 그냥 거래소 아니냐? 게임 도메인은 어디 있나?**
> 핵심은 거래소가 맞습니다 — 레이드는 그 경제에 **수요·공급을 만드는 엔진**이자, 서버 관점에선 또 하나의
> **원자 정산 도메인**입니다(거래소 프리미티브를 그대로 재사용). 인벤토리를 **스태시(안전, 12×가변)
> + 주머니** 그리드에 **장비 슬롯**(헬멧·방어구·무기 단일 슬롯 + 착용 시 내부 그리드를 여는 중첩
> 리그·백팩)을 더하고, 루프를 **서비스 계층 세션 상태머신**으로 모델링했습니다:
> `(없음) --start--> ACTIVE --extract--> EXTRACTED / --die--> DIED`. 핵심 설계는 세 가지입니다.
> ① **at-risk 자산 잠금**: 출격(`start`)은 스태시 밖 전부(주머니·장비·중첩 내용물)를 "위험" 상태로 잠그는데, 이건 새 메커니즘이
> 아니라 **매도 에스크로의 자산 잠금(스택 차감 / 인스턴스 owner=NULL)을 그대로 재사용**한 겁니다 —
> 덕분에 레이드 중 그 아이템의 판매/이동이 기존 에스크로 검사에 의해 자동 거부됩니다. ② **원자
> 정산**: `extract`(반입 자산을 **출격 시점 배치 그대로 제자리 복원** + 전리품 귀속) / `die`(위험분
> 전량 소각, 스태시는 불가침)를 각각 **단일 Postgres 트랜잭션**으로 처리해 총량 보존을 보장합니다.
> 제자리 복원은 `start` 시 각 반입 아이템의 원위치(컨테이너·슬롯·좌표)를 스냅샷해두고 되돌리는
> 방식입니다(자동 스태시 덤프가 아님). ③ **서비스 계약**: 이 백엔드는 게임 서버가
> 호출하는 서비스이고 `extract`/`die`는 명시적 엔드포인트입니다 — 실제 통합에서는 게임플레이 틱/전투를
> 도는 **게임 서버가 결과를 알립니다**(전투·실제 루팅은 범위 밖, 현재 `loot`는 MVP 시뮬레이션).
> 즉 레이드는 거래소 경제의 **공급·소각 밸브**(사망 소실이 마켓 희소성을, 탈출분이 매물을 만든다)이고,
> 아이템 이동은 전부 append-only **`item_ledger`(프로버넌스 회계)**로 기록해 감사 가능합니다.
> 플레이어당 ACTIVE 세션은 1개(부분 유니크 인덱스로 DB가 최종 강제)이고, grain 단일 활성화로
> start/extract/die가 자동 직렬화됩니다. 총량 보존·스태시 불가침·이중 출격 거부를 **불변식 통합
> 테스트**로 고정했습니다.

**Q. 인기 종목 하나에 트래픽이 몰리면? (핫 grain)**
> 단일 종목은 grain 하나에 직렬화되어 처리량 상한이 생깁니다. 이건 결함이 아니라 **정확성-처리량
> 트레이드오프**이고, 실제 거래소도 상품당 매칭 스레드가 1개입니다. 뚫으려면 **가격밴드 샤딩**으로
> `(templateId, priceBand)` grain을 나눠 병렬화하는데, 그 대가로 **엄격한 전역 가격-시간 우선을
> 포기**(밴드 격리 매칭)해야 합니다 — 단일 종목의 엄격한 우선순위는 원리적으로 단일 직렬화 지점을
> 요구하기 때문입니다. opt-in으로 구현해 핫 종목을 2.2배(299→649 orders/s) 뚫었고, 트레이드오프를
> 테스트로 명시했습니다. "무지성 샤딩"이 아니라 뭘 포기하는지 아는 게 포인트입니다.

**Q. 스케일 아웃은?**
> Orleans adonet 클러스터링(Postgres 멤버십)으로 실로를 늘리면 서로 다른 아이템 grain이 물리적으로
> 분산됩니다. 실시간은 Redis 백플레인으로 인스턴스 간 푸시를 중계합니다. 최종 병목은 단일 Postgres가
> 되고, 그 다음 단계는 읽기 리플리카/샤딩입니다.

**Q. grain끼리 호출하는데 데드락 안 나나?**
> OrderBook → Wallet/Inventory **단방향**이라 사이클이 없습니다. 순환 호출이 생기면 reentrancy를
> 검토해야 하는데, 가격밴드 코디네이터는 무상태라 조건부 reentrant로 두어 새 병목이 안 되게 했습니다.

## 6. 한계 · 다음 스텝 (자기인식 어필)

솔직하게 말할 수 있어야 신뢰를 줍니다:
- **초점**: 이 프로젝트의 중심은 **아웃게임 거래소**입니다. 레이드는 경제에 수요/공급을 만드는 얇은
  엔진으로 유지하며(실제 전투/루팅은 서버 드롭테이블 시뮬레이션), 게임성 자체를 깊게 파지 않았습니다 —
  깊이의 우선순위를 매칭·정산·동시성에 두었습니다.
- **인증**: 데모 로그인은 비밀번호가 없습니다(포트폴리오 스코프). 실서비스면 자격증명·비대칭키·시크릿
  매니저·토큰 폐기 리스트(jti 클레임은 이미 넣어둠)가 필요합니다.
- **관측성**: 구조적 로깅은 있으나 메트릭/트레이싱(OpenTelemetry)은 로드맵. 운영 대시보드는 경량으로 얹을 계획.
- **경제 엔진(구현 완료)**: 익스트랙션 세션 루프(반출→생존시 제자리 복원/사망시 소실→마켓 유입),
  그리드 인벤토리 + 장비 슬롯(헬멧·방어구·무기 + 중첩 리그·백팩), 아이템 프로비넌스 원장
  (`item_ledger`)을 붙여 거래소 경제의 공급·소각 밸브로 연결했습니다. 남은 부분은 보험, 실제 전투/루팅
  (현재 `loot`는 MVP 시뮬레이션), 그리드 회전입니다.
  이상거래(RMT) 탐지는 지갑/아이템 원장 기반으로 확장 가능합니다.
- **단일 노드 측정**: 성능 수치는 로컬 단일 노드 기준이라 절대치보다 상대 비교·불변식이 핵심 산출물입니다.

## 7. 화이트보드용 30초 아키텍처

```
Vue(SPA) ─REST/JWT─▶ ASP.NET Core API ─┬─ Orleans grains(매칭/지갑/인벤/스태시/레이드)
         ◀─SignalR(실시간)─┘            └─ Dapper 단일 트랜잭션 ─▶ Postgres(진실)
  다중 인스턴스: Orleans 클러스터링(Postgres 멤버십) + SignalR Redis 백플레인
```
말할 순서: ① 요청 흐름 → ② grain 단일 활성화로 매칭 직렬화 → ③ 정산 원자성 → ④ 인메모리는 투영,
DB가 진실(재수화) → ⑤ 스케일 아웃(실로+Redis).

## 8. 예상 꼬리질문 체크리스트
- "부분 체결은 어떻게 처리?" → 잔량을 호가창에 잔존, 상태 PARTIALLY_FILLED, 남은 물량은 다음 매칭 대상.
- "체결가는 누구 가격?" → 메이커(호가창 잔존 주문) 가격, 테이커에게 차익 환불.
- "자전거래 막나?" → 매칭 루프에서 본인 주문 스킵(수수료로 체결량 조작 방지), 교차 주문은 공존.
- "재시도로 주문 두 번 들어가면?" → `Idempotency-Key`로 (player,key) 슬롯을 **Redis `SET NX EX`** 선점,
  중복은 저장된 응답 반환(원본 처리 중이면 409). 미구성 시 무저장 폴백.
- "남용 방어?" → 플레이어별 레이트 리미팅, 429 + 표준 봉투 + Retry-After.
- "테스트 전략?" → 단위(순수 로직) + 통합(Testcontainers로 실제 DB/호스트) 우선, 125개, 동시성·불변식 포함.
